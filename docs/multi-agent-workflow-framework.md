# Multi-Agent 协同工作流框架

## 一、背景与目标

**解决的问题**：减少人工控制次数，实现自动化闭环工作流

**核心目标**：
- 用户提出需求后，自动拆解并确认
- 确认后自主调度执行
- 执行后自动复盘审查
- Agent 间互相验证、监督，形成自循环

---

## 二、Agent 角色定义

| 角色 | 数量 | 职责 | 决策权 |
|------|------|------|--------|
| **Master** | 1 | 接收需求、协调调度、用户交互、流程控制 | 高 |
| **Architect** | 1 | 任务拆解、标签标注、专业匹配、并行规划、API结构审查 | 高 |
| **Coder** | N | 业务代码实现 | 低 |
| **Researcher** | N | 技术调研、方案分析 | 低 |
| **Tester** | N | 测试编写与执行 | 低 |
| **DevOps** | N | 部署、CI/CD、环境配置 | 低 |
| **DataProcessor** | N | 数据处理任务 | 低 |
| **SecurityExpert** | N | 安全相关任务 | 低 |
| **PerformanceEngineer** | N | 性能分析、瓶颈定位、优化建议 | 低 |
| **IntegrationEngineer** | N | API设计、接口规范、系统集成 | 低 |
| **UXUIDesigner** | N | 界面设计、用户体验优化 | 低 |
| **Reviewer** | 1 | 多维度质量评分（权重 0.7） | 否决权 |
| **Reflector** | 1 | 复盘分析、流程改进建议（权重 0.3） | 建议权 |
| **Docer** | 1 | 技术文档生成 | 低 |

## 三、动态匹配机制

### 3.1 任务标签 → 执行者映射

| 任务标签 | 匹配执行者 | 并行上限 |
|----------|------------|----------|
| `[data]` | DataProcessor | 3 |
| `[security]` | SecurityExpert | 3 |
| `[test]` | Tester | 3 |
| `[deploy]` | DevOps | 3 |
| `[research]` | Researcher | 3 |
| `[performance]` | PerformanceEngineer | 3 |
| `[integration]` | IntegrationEngineer | 3 |
| `[ui]` / `[ux]` / `[design]` | UXUIDesigner | 3 |
| `[default]` | Coder | 3 |

### 3.2 匹配规则

- Architect 拆解任务时，根据任务类型打标签
- 同一标签任务可并行执行（上限3个）
- 未匹配任务默认指派给 Coder

### 4.2 API 设计流程（前置）

当项目涉及 API 开发或模块间接口定义时，必须在 Coder 开始编码前完成 API 设计。

#### 4.2.1 IntegrationEngineer 职责

- 根据 Architect 分配的模块，设计各模块的输入输出约束
- 定义接口规范（RESTful、参数、返回值）
- 制定数据结构定义
- 编写 API 契约文档

#### 4.2.2 Architect 审查要点

- API 是否符合 RESTful 规范
- 模块间接口是否一致
- 输入输出约束是否完整
- 错误处理是否合理
- 是否存在循环依赖

#### 4.2.3 约束

- **禁止** Coder 在 API 契约未确认前开始编码
- **禁止** IntegrationEngineer 设计未经验证的接口
- **禁止** Architect 跳过 API 审查环节

### 4.3 执行循环详情

| 迭代 | 动作 |
|------|------|
| 1 | 执行者完成任务 → Reviewer评分 → Reflector复盘 |
| 2 | 如评分<90，根据建议改进 → 重新评分 |
| 3 | 最后一次尝试机会，仍不达标则终止 |

## 五、评分机制

### 5.1 多维度评分表

| 维度 | 说明 | 满分 |
|------|------|------|
| 正确性 | 代码逻辑正确、符合需求 | 25 |
| 可读性 | 代码/文档结构清晰、易维护 | 20 |
| 完整性 | 功能覆盖完整、边界处理 | 20 |
| 性能 | 效率、资源占用合理 | 15 |
| 安全性 | 无明显安全风险 | 10 |
| 规范性 | 符合项目惯例、最佳实践 | 10 |

### 5.2 加权计算

```
最终评分 = Reviewer得分 × 0.7 + Reflector得分 × 0.3
```

### 5.3 终止条件

- **通过条件**：最终评分 ≥ 90
- **强制终止**：达到 3 次迭代

---

## 六、自循环机制

### 6.1 循环定义

Agent 间互相验证、监督：
- Executor → Reviewer → 评分反馈
- Reviewer → Reflector → 改进建议
- Reflector → Executor → 迭代依据

### 6.2 主动求助

**触发条件**：连续 2 次迭代无显著进展（评分提升 < 5）

**处理流程**：
```
触发 → Master 暂停工作流 → 向用户报告当前状态 → 等待用户指示
```

## 七、状态持久化

### 7.1 文件目录结构

```
.agent_workflow/
├── meta.md              # 工作流元数据（任务ID、创建时间、状态）
├── task_list.md         # Architect 输出：模块级任务清单
├── execution_plan.md    # 执行计划（含并行分配方案）
├── context.md           # 共享上下文（当前状态、历史决策）
├── scores.md            # 评分记录
├── iterations/
│   ├── iteration_1.md
│   ├── iteration_2.md
│   └── iteration_3.md
└── output/
    ├── final_code       # 最终代码产出
    └── final_doc        # 最终文档产出
```

### 7.2 文件说明

| 文件 | 维护者 | 更新时机 | 内容 | 操作约束 |
|------|--------|----------|------|----------|
| meta.md | Master | 每次迭代开始/结束 | 工作流整体状态、开始/结束时间 | 必须更新当前迭代和状态字段 |
| task_list.md | Architect | 任务拆解完成后 | 拆解后的模块任务列表 | 禁止修改已确认的计划 |
| execution_plan.md | Architect | 任务拆解完成后 | 任务分配、并行策略、执行顺序 | 禁止修改已确认的计划 |
| context.md | 各Agent | 任务交接/状态变更 | 共享信息传递 | 仅追加自身状态，禁止修改其他Agent内容；必须立即更新 |
| scores.md | Master | 每次评分后 | 每轮迭代的评分明细及最终得分 | 仅Master可写 |
| iterations/N.md | Master统筹 | 迭代结束后 | 每次迭代的详细记录 | Master写入执行记录，Reviewer写入评分区域，Reflector写入复盘区域 |
| workflow_changelog.md | Master | 每次工作流行为后 | 工作流历史行为记录 | 仅Master可写，Agent通过Master提取关键信息 |
| output/final_* | Docer/Coder | 任务完成后 | 最终产出物 | 禁止写入workflow_changelog.md |

---

### 7.3 文档操作约束

#### 7.3.1 通用约束（所有Agent）
- **禁止**修改其他Agent在context.md中的内容，仅追加自身状态
- **禁止**延迟更新状态文件，必须在操作完成后立即更新
- **禁止**写入workflow_changelog.md（仅Master可写）

#### 7.3.2 Master特殊约束
- 必须在任务交接前验证状态文件一致性
- 必须在每次迭代开始/结束时更新meta.md的"当前迭代"和"状态"字段
- 必须维护workflow_changelog.md，记录每次工作流行为

#### 7.3.3 iteration_N.md区域划分
1. **执行记录区域**：Master写入，记录任务执行过程
2. **评分区域**：Reviewer写入，包含各维度评分和改进建议
3. **复盘区域**：Reflector写入，包含流程分析和改进建议
各区域互不覆盖，由相应agent负责维护

#### 7.3.4 验证机制
- Master在任务交接前必须验证状态文件一致性
- 状态文件时间戳与实际执行时间偏差超过5分钟时，触发警告
- 警告后必须补全文档后才能继续执行

## 八、用户干预点

| 时机 | 说明 |
|------|------|
| 计划确认 | Architect 拆解完成后，用户确认执行计划 |
| 主动求助 | 连续2次无进展时，用户介入决策 |

**设计原则**：用户只在必要时介入，避免频繁打断

---

## 九、关键约束

1. **执行者并行上限**：同类执行者最多 3 个并行实例
2. **迭代上限**：3 次，强制终止
3. **评分权重**：Reviewer 70% + Reflector 30%
4. **任务粒度**：Architect 拆解到模块级，不拆解到具体代码行
5. **状态持久化**：所有关键状态存入 .agent_workflow/ 目录
6. **API 设计前置**：涉及 API/接口的任务必须先完成 API 设计流程
7. **禁止跳过审查**：API 设计必须经过 Architect 审查才能进入编码阶段
8. **文档操作约束**：所有Agent必须遵守文档实时性约束，禁止修改其他Agent内容，必须立即更新状态文件
