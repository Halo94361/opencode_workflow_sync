## Skill 写作指导助手

### 概述

指导新 Skill 的编写与优化。遵循四大核心原则：

1. **Description 是触发条件**：告诉模型何时该调用，而非功能简介
2. **渐进式披露**：元数据 → 正文 → 资源文件，分层加载
3. **最小化工具权限**：`allowed-tools` 只包含必要工具
4. **Gotchas 要真实**：来自实践的踩坑点，不是臆想

### 前置要求

- 输入：提供新 Skill 的需求描述或现有 Skill 文件
- 权限：创建和修改 `.opencode/skills/` 目录下的文件
- 环境：掌握 OpenCode Agent & Skill 机制

### 执行流程

| 步骤 | 名称 | 输入 | 输出 | 说明 |
|------|------|------|------|------|
| 1 | 理解意图 | 用户需求描述 | 需求收集表 | 明确功能、触发条件、期望输出 |
| 2 | 需求分析 | 需求收集表 | Skill 类型、目录结构、工具权限 | 判断 Skill 类型，规划目录结构 |
| 3 | 编写 SKILL.md | 需求分析结果 | SKILL.md 文件 | 填写 Frontmatter，编写正文 |
| 4 | 质量检查 | SKILL.md 文件 | 检查报告 | 对照清单逐项检查 |
| 5 | 验证迭代 | 检查报告 | 改进后的 SKILL.md | 设计测试用例，迭代优化 |

### 输出格式

生成符合以下标准结构的 SKILL.md 文件：

```markdown
---
name: skill-name
description: 当用户要求[动作]或提出[关键词]时触发。适用于[场景]，不适用于[排除场景]。
allowed-tools:
  - read
  - write
version: 1.0.0
---
```

### Gotchas（踩坑点）

- ⚠️ **Description 过于通用**：描述太模糊会导致不触发或误触发，例如"代码工具"会匹配太多场景。必须明确触发动作和排除场景。
- ⚠️ **工具权限过大**：列出不必要的工具可能让模型执行意外操作。遵循最小权限原则，只包含必要工具。
- ⚠️ **引用未验证的假设**：不要在 Skill 中假设用户环境已配置某工具。需要时在「前置要求」中说明。
- ⚠️ **路径硬编码**：Windows 路径 `D:\` 在 Linux 上无效，必须使用 `{baseDir}` 引用路径。
- ⚠️ **正文结构不规范**：不使用标准结构会导致模型难以定位信息。遵循「前置要求 → 执行流程 → 输出格式 → Gotchas → 术语表」顺序。
- ⚠️ **缺少术语表**：专业术语需要在术语表中定义，避免模型理解偏差。
- ⚠️ **选项命名冲突**：模板中不同功能使用相同短选项（如 `-v` 同时用于 `--version` 和 `--verbose`）会导致用户混淆。使用前检查所有选项命名。
- ⚠️ **Markdown 代码块嵌套**：SKILL.md 中展示模板内容时，避免在代码块内嵌套代码块（如 ````markdown` 内包含 `````bash`），会导致渲染错误。模板内容应放在独立文件中。
- ⚠️ **模板未使用导入**：模板代码中的未使用导入（如 Python 的 `import os`）会降低代码质量。移除或注释并添加说明。
- ⚠️ **占位符标记不清晰**：模板中的示例数据（如 `your-repo`、`example@xxx.com`）需明确标记为占位符，避免用户直接使用。使用 `<your-xxx>` 或 `{{placeholder}}` 格式。

### 术语表

| 术语 | 说明 |
|------|------|
| Frontmatter | SKILL.md 顶部的 YAML 元数据区域，定义 name、description、allowed-tools 等 |
| Skill | AI Agent 的能力定义单元，通过 YAML + Markdown 向模型传递能力边界 |
| Gotcha | 踩坑点，已知的陷阱和注意事项，来自实践经验 |

| 渐进式披露 | 元数据 → 正文 → 资源文件，分层加载，按需获取信息 |
| 最小权限原则 | 只授予完成任务所需的最少工具权限 |

## 扩展参考

### 阶段一：理解意图（Capture Intent）

与用户沟通，明确以下要素：

| 问题 | 说明 | 示例 |
|------|------|------|
| 这个 Skill 做什么？ | 核心能力边界 | "分析代码质量、发现潜在Bug" |
| 何时触发？ | 触发条件 | "用户说'帮我审查代码'、'看看有什么问题'" |
| 期望输出是什么？ | 输出格式 | "问题列表 + 严重度分级 + 修复建议" |
| 需要测试吗？ | 验证需求 | "需要真实场景验证" |

### 阶段二：需求分析（Research & Analysis）

#### 1. Skill 类型判断

| 类型 | 特征 | 触发词 |
|------|------|--------|
| 诊断类 | 错误分析、问题定位 | "诊断"、"这是什么错"、"帮我看看" |
| 生成类 | 代码/文档/测试生成 | "帮我写"、"生成" |
| 审查类 | 质量评估、安全审计 | "审查"、"评估"、"分析" |
| 工作流类 | 多步骤协作、迭代执行 | "规划"、"协同"、"工作流" |

#### 2. 目录结构规划

```
skill-name/
├── SKILL.md              # 【必须】核心定义
├── scripts/              # 【可选】可执行脚本
├── references/           # 【可选】参考文档
├── assets/               # 【可选】模板资源
└── logs/                 # 【可选】运行日志
```

#### 3. 工具权限评估

| 工具 | 适用场景 | 排除场景 |
|------|---------|---------|
| read | 读取代码/文档 | 无 |
| write | 生成新文件 | 已有文件应使用edit |
| edit | 修改现有文件 | 生成新文件 |
| glob | 文件模式匹配 | 精确路径用read |
| grep | 内容搜索 | 文件存在性用glob |
| bash | 执行脚本、系统命令 | 纯逻辑操作 |

### 阶段三：编写 SKILL.md（Write SKILL.md）

#### 1. YAML Frontmatter 模板

```yaml
---
name: skill-name
description: 当用户要求[动作]或提出[关键词]时触发。适用于[场景]，不适用于[排除场景]。
allowed-tools:
  - read
  - write
version: 1.0.0
---
```

#### 2. Description 写作规范

**核心公式**：
```
当用户要求[动作]或提出[关键词]时触发。适用于[语言/框架/场景]，不适用于[排除场景]。
```

**示例对比**：

| ❌ 错误 | ✅ 正确 |
|--------|--------|
| `代码审查技能` | `当用户要求代码审查、性能分析、安全审计或提出"SOLID原则"时触发。适用于Python/Java/Go项目，不适用于架构设计评审。` |
| `API文档生成` | `当用户提供头文件、函数签名或接口描述并要求生成API文档时触发。适用于C/REST API场景，不适用于已提供完整Swagger定义。` |

### 阶段四：质量检查（Quality Check）

#### 发布前检查清单

| 检查项 | 要求 | 状态 |
|--------|------|------|
| Description | 触发条件写法，50-200字符 | □ |
| Allowed-tools | 最小化，仅必要工具 | □ |
| Gotchas | 至少3个真实踩坑点 | □ |
| 路径引用 | 使用`{baseDir}`，无硬编码 | □ |
| 语气 | 祈使语气，无"你应该" | □ |
| 版本号 | 语义化版本（x.y.z） | □ |
| 字数 | 正文 < 5,000字 | □ |
| 选项命名 | 无冲突，不同功能用不同短选项 | □ |
| 代码块嵌套 | SKILL.md 中避免嵌套 markdown 代码块 | □ |
| 模板导入 | 移除未使用的 import 语句 | □ |
| 占位符标记 | 使用 `<your-xxx>` 或 `{{placeholder}}` 格式 | □ |

#### Description 自检问题

- [ ] 包含触发动作（"当用户要求..."）
- [ ] 包含触发关键词（"或提出'xxx'时"）
- [ ] 明确适用场景（"适用于..."）
- [ ] 明确排除场景（"不适用于..."）
- [ ] 无硬编码路径或绝对路径

#### 迭代改进

| 迭代目标 | 改进方向 |
|----------|----------|
| 触发不准 | 优化description，覆盖更多同义表达 |
| 输出不符合预期 | 完善输出格式示例，增加具体指引 |
| 遗漏边界场景 | 补充Gotchas，扩展排除场景 |
| 工具权限不当 | 调整allowed-tools列表 |

## 常见错误与修正

| 错误类型 | 问题 | 修正方式 |
|----------|------|----------|
| Description太模糊 | `代码分析工具` | 明确触发动作和场景 |
| 工具权限过大 | `allowed-tools: [all]` | 仅列出必要工具 |
| Gotchas缺失 | 无踩坑点 | 从实践中总结真实陷阱 |
| 路径硬编码 | `D:\projects\...` | 使用 `{baseDir}` |
| 语气不当 | `你应该先分析...` | 改为祈使语气 |
| 选项命名冲突 | `-v` 同时用于 version 和 verbose | 使用 `-V` 用于 version，`-v` 用于 verbose |
| 代码块嵌套 | ````markdown` 内嵌套 ````bash` | 将模板内容放在独立文件中 |
| 未使用导入 | 模板中有 `import os` 但未使用 | 移除或注释并说明用途 |
| 占位符不清 | `your-repo`、`example@xxx.com` | 改为 `<your-repo>`、`<email@example.com>` |

|----------|----------|
