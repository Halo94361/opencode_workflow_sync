# 执行循环详细说明

---

## 一、执行循环概述

```
用户需求 → Master → 项目探索 → Architect拆解 → 用户确认 → 执行循环(最多3次)
                                    ↓
                   Reviewer评分(0.7) + Reflector复盘(0.3)
                                    ↓
                          评分≥90? → [是]终止 [否]迭代
                                    ↓
                          (需文档时) → Docer → Reviewer审阅
```

---

## 二、项目探索流程

### 2.1 触发条件

自动检测是否为已有项目：

- 项目目录存在非空 src/、lib/、drivers/、app/ 等目录
- 存在构建文件（Makefile/CMakeLists.txt/build.gradle/setup.py等）
- 存在版本控制（.git目录）
- 用户未明确说明是新项目

### 2.2 Master 项目检测步骤

```markdown
## 项目检测报告

### 检测结果
- Is Existing Project: true/false
- 项目路径: <路径>
- 项目规模: 小型/中型/大型

### 探索触发
- 触发: 是/否
- 原因: <触发或跳过原因>
```

### 2.3 ProjectExplorer 执行步骤

#### 步骤1：项目结构分析

- 读取项目根目录结构
- 识别源代码目录（src/, lib/, app/, drivers/等）
- 识别配置文件目录
- 识别文档和测试目录

#### 步骤2：技术栈识别

- 识别编程语言（通过文件扩展名）
- 识别框架和库（通过依赖文件）
- 识别构建工具

#### 步骤3：模块关系分析

- 分析模块间依赖关系
- 识别核心模块和边界模块
- 识别入口文件

#### 步骤4：输出探索报告

### 2.4 大型项目分段探索

根据项目规模，采用不同策略：

| 项目规模 | 文件数量 | 策略 |
|----------|----------|------|
| 小型 | <50 | 单一探索者完整探索 |
| 中型 | 50-200 | 按模块分配多个探索者 |
| 大型 | >200 | 核心模块优先，渐进探索 |

### 2.5 探索报告格式


## 三、Master 调度步骤

### 3.1 前置条件

- 用户已确认执行计划
- 项目探索已完成（如需）

### 3.2 调度原则

- 同一标签任务可并行执行（上限3个）
- 独立任务优先调度
- 依赖任务按顺序调度

### 3.3 执行动作

#### 步骤1：解析执行计划

- 读取 execution_plan.md
- 识别可并行任务组
- 确定任务依赖关系

#### 步骤2：分配执行者

- 按标签匹配执行者
- 检查并行上限

#### 步骤3：监控执行

- 收集执行结果
- 处理异常情况

### 3.4 输出格式

```markdown
## 调度报告

### 任务分配
| 任务 | 执行者 | 状态 | 备注 |
|------|--------|------|------|
| T1 | Coder | pending | |
| T2 | Researcher | running | |
| T3 | Coder | pending | 依赖T1 |
```

## 四、Architect 任务拆解步骤

### 4.1 前置条件

- Master 调用
- 项目探索已完成（如需）

### 4.2 执行动作

#### 步骤1：分析需求

- 理解业务目标
- 识别核心功能

#### 步骤2：拆解任务

- 按模块拆解
- 标注任务标签
- 确定并行组

#### 步骤3：输出文档

- task_list.md
- execution_plan.md

## 五、执行者执行步骤

### 5.1 前置条件

- Master 分配任务

### 5.2 执行动作

#### 步骤1：读取上下文

- 读取 execution_plan.md
- 读取 context.md（共享状态）
- 读取 project_exploration.md（如有）

#### 步骤1.5：路径解析（新增）

- 确认任务 Input/Output 中的路径格式：
  - 仓库内文件：应为 `src/...`、`lib/...` 等相对路径
  - 仓库外文件：应为 `C:\...`、`D:\...`、`/home/...` 等绝对路径
- 如发现裸文件名（如 `utils.py` 而非 `src/utils.py`）：
  - 优先从 project_exploration.md 的模块分析中查找匹配文件
  - 如找不到，列出可能位置向 Master 求助
- 将解析后的完整路径映射存入 context.md：

```markdown
## 当前任务路径映射
- T1 Input: src/utils/parser.py (已修正自 utils.py)
- T1 Output: src/utils/validator.py
```

#### 步骤2：执行任务

- 按质完成指派任务
- 记录执行进度

#### 步骤3：更新状态

- 更新 context.md
- 如实报告给 Master

### 5.3 输出格式

```markdown
## 执行报告

### T1 - 任务名
- Status: completed / failed
- 执行者: <name>
- 耗时: <时间>
- Input: <实际使用的完整路径>
- Output: <实际使用的完整路径>
- 问题: <如有>

### T2 - 任务名
- Status: completed
- 执行者: <name>
- 耗时: <时间>
- Input: <实际使用的完整路径>
- Output: <实际使用的完整路径>
```

## 六、主动求助

### 触发条件

连续 2 次迭代无显著进展（评分提升 < 5）

### 处理流程

```
触发 → Master 暂停工作流 → 向用户报告当前状态 → 等待指示
```

### 报告内容

```markdown
## 主动求助报告

### 当前状态
- 迭代次数: 2
- 当前得分: 82
- 上次得分: 80
- 评分提升: 2 (< 5)

### 已完成
- T1: 完成
- T2: 完成
- T3: 进行中

### 问题分析
<问题描述>

### 需要用户决策
1. 是否继续迭代？
2. 是否调整任务范围？
3. 是否需要人工介入？
```

## 七、迭代流程

### 7.1 迭代详情

| 迭代 | 动作 |
|------|------|
| 1 | 执行者完成任务 → Reviewer评分 → Reflector复盘 |
| 2 | 如评分<90，根据建议改进 → 重新评分 |
| 3 | 最后一次尝试机会，仍不达标则终止 |

### 7.2 迭代判断

```markdown
## 迭代判断

### 评分计算
最终得分 = Reviewer得分 × 0.7 + Reflector得分 × 0.3

### 判断结果
- 最终得分 >= 90: 终止（通过）
- 最终得分 < 90 且 迭代 < 3: 继续迭代
- 最终得分 < 90 且 迭代 = 3: 终止（未通过）
```

---

## 八、执行状态流转

### 8.1 任务状态

```
PENDING → IN_PROGRESS → COMPLETED
                      ↘ FAILED
```

### 8.2 工作流状态

```
IN_PROGRESS → COMPLETED (评分>=90)
            ↘ TERMINATED (迭代达上限或用户终止)
```
