# 项目探索流程详细说明

---

## 一、流程概述

当检测到是已有项目时，自动触发项目探索。

### 流程图

```
用户需求
    │
    ▼
┌─────────────────────────────┐

│  检测是否为已有项目           │

│  - 新项目: 跳过探索         │
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
```

## 二、触发条件

### 2.1 自动检测

当满足以下条件时，自动触发项目探索：

- 项目目录存在非空 src/、lib/、drivers/、app/、components/ 等源代码目录
- 存在构建文件（Makefile/CMakeLists.txt/build.gradle/setup.py/Cargo.toml等）
- 存在版本控制（.git目录）
- 用户未明确说明是新项目

### 2.2 检测步骤

```markdown
## 项目检测报告

### 检测结果
- Is Existing Project: true/false
- 项目路径: <路径>
- 项目规模: 小型/中型/大型

### 探索触发
- 触发: 是/否
- 原因: <触发或跳过原因>
```

## 三、ProjectExplorer 执行步骤

### 3.1 步骤1：项目结构分析

- 读取项目根目录结构
- 识别源代码目录（src/, lib/, app/, drivers/, components/等）
- 识别配置文件目录
- 识别文档和测试目录

### 3.2 步骤2：技术栈识别

- 识别编程语言（通过文件扩展名）
- 识别框架和库（通过依赖文件）
- 识别构建工具

**语言识别规则**：

| 扩展名 | 语言 |
|--------|------|
| .py | Python |
| .c/.h | C |
| .cpp/.hpp | C++ |
| .rs | Rust |
| .go | Go |
| .js/.ts | JavaScript/TypeScript |
| .java | Java |
| .kt | Kotlin |
| .swift | Swift |
| .m | Objective-C/ MATLAB |

**框架识别**：

| 文件 | 框架 |
|------|------|
| requirements.txt, setup.py | Python |
| package.json | Node.js |
| go.mod | Go |
| Cargo.toml | Rust |
| CMakeLists.txt | CMake/嵌入式 |
| Makefile | C/嵌入式 |
| build.gradle, settings.gradle | Android/Gradle |

### 3.3 步骤3：模块关系分析

- 分析模块间依赖关系
- 识别核心模块和边界模块
- 识别入口文件

### 3.4 步骤4：输出探索报告

## 四、大型项目分段探索

### 4.1 项目规模判断

| 项目规模 | 文件数量 | 策略 |
|----------|----------|------|
| 小型 | <50 | 单一探索者完整探索 |
| 中型 | 50-200 | 按模块分配多个探索者 |
| 大型 | >200 | 核心模块优先，渐进探索 |

### 4.2 分段策略

- 按模块或目录划分探索区域
- 每个探索者负责指定区域
- 最终合并为完整探索报告

### 4.3 防止上下文溢出

- 控制单次探索的文件数量
- 优先探索核心模块
- 大文件进行摘要而非完整读取

## 八、增量更新机制

### 8.1 维护目标
- 避免每次新对话都重新探索项目
- 项目发生变动时主动更新报告
- 保持报告与实际项目同步

### 8.2 维护触发条件
| 触发时机 | 触发者 | 动作 |
|----------|--------|------|
| 工作流启动时 | Master | 检查文件存在性和 git_head |
| 执行者任务完成时 | Master | 检测变更，决定是否触发更新 |
| 手动 [explore] 标签 | 任意 | 全面重扫 |

### 8.3 变动检测方法
```bash
# 检测工作区变动
git status --porcelain

# 对比 HEAD 与工作区
git diff --name-only HEAD

# 获取最近一次 commit 的文件变更
git diff --name-only HEAD~1 HEAD
```

### 8.4 变更阈值判定
| 变更类型 | 判断标准 | 处理 |
|----------|----------|------|
| 显著变更 | 新增/删除/重命名目录，或 >阈值文件变更 | 触发增量更新 |
| 微小变更 | ≤阈值文件变更且非目录级 | 记录日志，暂不更新 |
| 配置变更 | 只涉及配置文件（.yaml/.json/.toml） | 记录日志，暂不更新 |

**阈值配置（按项目规模）**：
| 项目规模 | 文件数量 | 文件阈值 | 目录阈值 |
|----------|----------|----------|----------|
| 小型 | <50 | >3 | ≥1 |
| 中型 | 50-200 | >5 | ≥2 |
| 大型 | >200 | >10 | ≥3 |

### 8.5 更新类型

| 条件 | 更新方式 |
|------|----------|
| 项目首次探索 | 全量更新 |
| git diff 显示显著目录/文件变更 | 全量更新 |
| git diff 显示 ≤阈值文件变更 | 增量更新 |
| 无 git diff（新对话直接复用） | 无需更新 |

### 8.6 增量更新 vs 全量更新

**全量更新**：重新生成完整报告，version 主版本递增（如 v1.0 → v2.0）

**增量更新**：
1. 读取当前 `project_exploration.md`
2. 仅重新分析受影响的模块
3. 更新相关章节，保留未受影响内容
4. 追加增量更新记录到 `## 增量更新记录`（**仅保留最新 5 条**）
5. 更新元数据：`last_update`、`git_head`、`update_count`
6. version 次版本递增（如 v2.0 → v2.1）

### 8.7 读取范围限制

- **增量更新记录**：Agent 读取时**只读取最新一条**（首条），用于了解当前项目状态
- **完整变更日志**：仅供审计追溯，**禁止 Agent 读取**
- **原因**：避免上下文溢出和理解冲突

## 九、约束与处理

### 9.1 新增约束规则

| 约束 | 说明 |
|------|------|
| 复用优先 | 已有 `project_exploration.md` 时必须先读取复用，禁止直接全量探索 |
| 读取范围限制 | Agent 读取 project_exploration.md 时只读最新一条增量更新记录，禁止读完整变更日志 |
| 更新前置 | Coder 等修改代码后，Master 必须评估是否需要更新报告 |
| 增量原则 | 项目无显著结构变更时，应进行增量更新而非全量重写 |
| 日志追加 | 每次更新必须在增量更新记录中追加，禁止覆盖历史 |
| 版本递增 | 每次更新（无论增量还是全量）必须递增 `report_version` |
| 元数据同步 | 每次更新必须同步 `last_update` 和 `git_head` |
| 禁止降级 | 禁止将 `format_version` 或 `report_version` 降低 |

### 9.2 违反处理

| 约束 | 违反表现 | 处理方式 |
|------|----------|----------|
| 复用优先 | 跳过现有报告直接全量探索 | Master 叫停，重新读取复用 |
| 读取范围限制 | Agent 读取完整变更日志 | Master 监督，违者重新执行 |
| 版本递增 | 更新后版本未递增 | Master 叫停，要求修正 |
| 禁止降级 | 故意降低版本号 | Master 叫停，追究责任 |
