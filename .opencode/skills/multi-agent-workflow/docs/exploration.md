# 项目探索流程详细说明

---

## 一、流程概述

当检测到是已有项目时，自动触发项目探索。

### 流程图

```
用户需求
    │
    ▼
┌─────────────────────────────┐
│      Master                  │
│  检测是否为已有项目           │
│  - 检测到: 触发探索          │
│  - 新项目: 跳过探索         │
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│    ProjectExplorer           │
│    项目探索（前置流程）       │
│    - 结构分析               │
│    - 技术栈识别             │
│    - 依赖关系分析           │
│    - 架构审查               │
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│       Architect              │
│  根据探索报告分配探索任务     │
│  - 大型项目分段探索         │
│  - 防止上下文溢出           │
└─────────────┬───────────────┘
              │
              ▼
      Architect 任务拆解
```

---

## 二、触发条件

### 2.1 自动检测

当满足以下条件时，自动触发项目探索：

- 项目目录存在非空 src/、lib/、drivers/、app/、components/ 等源代码目录
- 存在构建文件（Makefile/CMakeLists.txt/build.gradle/setup.py/Cargo.toml等）
- 存在版本控制（.git目录）
- 用户未明确说明是新项目

### 2.2 检测步骤

```markdown
## 项目检测报告

### 检测结果
- Is Existing Project: true/false
- 项目路径: <路径>
- 项目规模: 小型/中型/大型

### 探索触发
- 触发: 是/否
- 原因: <触发或跳过原因>
```

## 三、ProjectExplorer 执行步骤

### 3.1 步骤1：项目结构分析

- 读取项目根目录结构
- 识别源代码目录（src/, lib/, app/, drivers/, components/等）
- 识别配置文件目录
- 识别文档和测试目录

### 3.2 步骤2：技术栈识别

- 识别编程语言（通过文件扩展名）
- 识别框架和库（通过依赖文件）
- 识别构建工具

**语言识别规则**：

| 扩展名 | 语言 |
|--------|------|
| .py | Python |
| .c/.h | C |
| .cpp/.hpp | C++ |
| .rs | Rust |
| .go | Go |
| .js/.ts | JavaScript/TypeScript |
| .java | Java |
| .kt | Kotlin |
| .swift | Swift |
| .m | Objective-C/ MATLAB |

**框架识别**：

| 文件 | 框架 |
|------|------|
| requirements.txt, setup.py | Python |
| package.json | Node.js |
| go.mod | Go |
| Cargo.toml | Rust |
| CMakeLists.txt | CMake/嵌入式 |
| Makefile | C/嵌入式 |
| build.gradle, settings.gradle | Android/Gradle |

### 3.3 步骤3：模块关系分析

- 分析模块间依赖关系
- 识别核心模块和边界模块
- 识别入口文件

### 3.4 步骤4：输出探索报告

## 四、大型项目分段探索

### 4.1 项目规模判断

| 项目规模 | 文件数量 | 策略 |
|----------|----------|------|
| 小型 | <50 | 单一探索者完整探索 |
| 中型 | 50-200 | 按模块分配多个探索者 |
| 大型 | >200 | 核心模块优先，渐进探索 |

### 4.2 分段策略

- 按模块或目录划分探索区域
- 每个探索者负责指定区域
- 最终合并为完整探索报告

### 4.3 防止上下文溢出

- 控制单次探索的文件数量
- 优先探索核心模块
- 大文件进行摘要而非完整读取

## 六、约束与处理

### 6.1 约束规则

| 约束 | 说明 |
|------|------|
| 探索前置 | 已有项目必须先完成探索 |
| 任务分配 | 大型项目分段探索，防止上下文溢出 |

### 6.2 违反处理

| 约束 | 违反表现 | 处理方式 |
|------|----------|----------|
| 探索前置 | 跳过探索直接开发 | Master 叫停，重新探索 |
| 任务分配 | 单次探索范围过大 | Architect 重新分配 |
