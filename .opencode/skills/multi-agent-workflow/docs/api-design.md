# API 设计流程详细说明

---

## 一、流程概述

当项目涉及 API 开发或模块间接口定义时，必须在 Coder 开始编码前完成 API 设计。

### 流程图

```
Architect 任务拆解
        │
        ▼
┌───────────────────────────────┐
│   IntegrationEngineer          │
│   模块接口设计                 │
│   - 输入/输出约束             │
│   - 接口规范                  │
│   - 数据结构定义             │
└───────────────┬───────────────┘
                │
                ▼
┌───────────────────────────────┐
│       Architect               │
│   API 结构审查               │
│   - 接口合理性               │
│   - 模块间协作               │
│   - 一致性检查               │
└───────────────┬───────────────┘
                │
                ▼
         Coder 开始编码
```

---

## 二、IntegrationEngineer 执行步骤

### 2.1 前置条件

- Architect 已完成任务拆解
- 任务被分配 [integration] 标签
- 模块职责已明确

### 2.2 执行动作

#### 步骤1：分析模块边界

- 阅读任务描述和模块职责
- 识别需要暴露的接口
- 确定模块间数据流

#### 步骤2：设计接口契约

- 定义输入/输出参数
- 制定数据结构
- 编写错误码规范

#### 步骤3：输出 API 文档

- 按指定格式输出
- 包含所有约束条件

### 2.3 API 契约输出模板

```markdown
## 模块 [X] API 契约

### 模块职责
<简要描述模块职责>

### 接口清单

| 接口名 | 方法 | 输入 | 输出 | 约束条件 |
|--------|------|------|------|----------|
| funcA | POST | {a: int, b: str} | {result: int} | a > 0, len(b) <= 100 |

### 数据结构

```python
@dataclass
class InputA:
    a: int  # 约束: a > 0
    b: str  # 约束: len(b) <= 100

@dataclass
class OutputA:
    result: int
```

### 错误码规范

| 错误码 | 说明 | HTTP状态码 | 处理方式 |
|--------|------|------------|----------|
| 1001 | 参数错误 | 400 | 返回错误信息 |
| 1002 | 认证失败 | 401 | 跳转登录 |
| 1003 | 权限不足 | 403 | 提示权限申请 |
| 5000 | 内部错误 | 500 | 记录日志 |

### 依赖接口

| 接口名 | 来源模块 | 用途 |
|--------|----------|------|
| getUserInfo | user-service | 获取用户信息 |
```

## 三、Architect API 审查步骤

### 3.1 前置条件

- IntegrationEngineer 已输出 API 契约
- 所有涉及 API 的模块契约已生成

### 3.2 审查动作

#### 合理性检查

- 接口是否符合单一职责原则
- 参数是否必要且最小化
- 接口命名是否清晰

#### 一致性检查

- 模块间接口是否匹配
- 数据类型是否一致
- 错误码是否统一

#### 安全性检查

- 是否有注入风险
- 认证授权是否考虑
- 敏感数据是否保护

### 3.3 审查报告模板

```markdown
## API 审查报告

### 审查结果：通过 / 需修改

### 问题清单

| 问题 | 严重程度 | 位置 | 建议 |
|------|----------|------|------|
| 参数校验缺失 | 高 | moduleA/funcA | 添加参数校验 |
| 命名不一致 | 中 | moduleB | 统一命名为 camelCase |

### 修改意见
<具体修改意见>
```

## 四、约束与处理

### 4.1 约束规则

| 约束 | 说明 |
|------|------|
| API 设计前置 | 涉及 API/接口的任务必须先完成 API 设计流程 |
| 禁止跳过审查 | API 设计必须经过 Architect 审查才能进入编码阶段 |

### 4.2 违反处理

| 约束 | 违反表现 | 处理方式 |
|------|----------|----------|
| API 设计前置 | Coder 提前开始 | Master 叫停，重新走流程 |
| 禁止跳过审查 | 跳过审查环节 | 返回审查环节 |

## 五、API 设计最佳实践

### 5.1 RESTful 规范

- 使用标准 HTTP 方法（GET/POST/PUT/DELETE）
- 资源命名使用名词复数形式
- 使用 HTTP 状态码表示结果

### 5.2 接口设计原则

- **单一职责**：每个接口只做一件事
- **最小化参数**：只传递必要参数
- **版本控制**：考虑 API 版本演进
- **一致性**：命名、错误码、返回格式保持一致

### 5.3 安全性考虑

- 参数校验，防止注入
- 认证授权机制
- 敏感数据加密传输
- 限流保护
